> **スキル名:** ceremonial-rescue-jp
> **バージョン:** 3.0
> **作成日:** 2026-03-22
> **対象プラットフォーム:** Agent Skills標準に対応する全てのアプリ・サービス（Claude Cowork、Claude Code、OpenClaw、Antigravity、その他互換プラットフォーム）
> **v3.0（現在）の主要変更:** ハイブリッド知識アーキテクチャ（不変知識は静的DB + 変動情報のみWeb検索）、references/を3ファイルに統合再構成、テスト134件全パス

## 1. コンセプト

### 1.1 設計思想

**"気まずい・失礼が怖い・何をすればいいかわからない" を、一言入力から即座に解消する。**

このスキルの本質は「文書作成ツール」ではない。冠婚葬祭で不安を感じている人に「これで大丈夫」という確信を、最短で届けることが目的である。文書はその確信を支える手段の一つに過ぎない。

### 1.2 スキル名の変更

旧名「冠婚葬祭まるごとガイド＆文書作成」→ 新名「**冠婚葬祭レスキュー**」

理由：ユーザーが求めているのは「ガイド」ではなく「救助」。パニック状態の人が一言で助けを求められる名前にする。英語ID: `ceremonial-rescue-jp`

### 1.3 解決する課題

日本人が他人の冠婚葬祭に際して直面する4つの苦痛：

1. **正解がわからない不安** — 金額・敬語・マナー・服装の「正解」が、立場・関係性・宗派・地域・タイミングで変わる
2. **情報が散在している** — ググっても断片的。10サイト巡回しても「自分のケース」の答えがない
3. **文面を書けない** — 弔電、欠勤メール、スピーチ、LINE返信。慣れない文面を短時間で書く恐怖
4. **その場で何を言えばいいかわからない** — 受付・焼香・遺族への声かけ。文書以前に「口頭の一言」が出てこない

### 1.4 ターゲットユーザー

- 20〜60代の日本在住の社会人全般
- 特に：冠婚葬祭の経験が少ない若手社会人、会社関係の対応に迷うビジネスパーソン
- ITリテラシーは問わない（自然言語で一言伝えるだけ）

### 1.5 スコープの明確な境界

**対応する：** 他人の冠婚葬祭に対する自分の対応（参列者・関係者として）
**対応しない：** 婚姻届手続き、相続・遺産分割、年金・保険手続き、税務助言、宗派の深い儀礼解説、法的効力のある文書作成

## 2. ユーザープロファイル自律探索（v3.0改訂）

### 2.1 なぜプロファイル探索が必要か

Agent Skills対応のアプリ・サービスでは、ユーザーが自身の情報をワークスペース内のファイルに保存していることが多い。ファイル名・配置場所・フォーマットはプラットフォームやユーザーの運用によって異なり、統一された標準は存在しない。

したがって、このスキルは **特定のファイル名やパスに依存せず、AIが自律的にワークスペースを探索してユーザー情報を発見する** 方式を採る。

この情報を活用すれば「質問ゼロ」で精度の高い対応セットを出力できる。逆にプロファイルを読まずに動くと、既にわかっている情報を改めて質問することになり、ユーザー体験が大幅に劣化する。

### 2.2 プロファイルから取得する情報

| 情報             | 用途                                         | 取得できなかった場合                             |
| ---------------- | -------------------------------------------- | ------------------------------------------------ |
| 氏名             | 文書の差出人名、署名に自動挿入               | 空欄にして「ここにお名前を入れてください」と注記 |
| 勤務先・所属部署 | 会社関係モードの自動判定、社内文書のヘッダー | 質問しない。必要時のみ確認                       |
| 役職             | 敬語レベルの調整、代表対応かどうかの判定     | デフォルト（一般社員想定）で出力                 |
| 居住地域         | 地域慣習の自動適用（関西/関東等）            | 全国標準で出力                                   |
| 家族構成         | 親族関係の理解、「義母」「義父」等の自動解釈 | 入力テキストから推定                             |
| 宗教・宗派       | 自分側の宗派情報（相手側とは別）             | 使用しない（相手側の宗派のみ考慮）               |
| メールアドレス   | メール文面の署名                             | 空欄で出力                                       |
| 電話番号         | 文書の署名                                   | 空欄で出力                                       |
| 過去の利用履歴   | 同じ相手への重複対応チェック                 | チェックなし                                     |

### 2.3 自律探索の手順

スキル起動時に、AIが以下のステップで自律的にユーザー情報を探索する。特定のファイル名をハードコードせず、ワークスペースの構造を読み取って判断する。

```
ステップ1：ワークスペースのディレクトリ一覧を取得する
  → ルート直下のファイル名・フォルダ名を一覧する

ステップ2：以下の手がかりから、ユーザー情報が含まれていそうなファイルを特定する
  手がかり（優先度順）：

  a. プラットフォームがユーザー情報用として定義しているファイル
     → OpenClaw: USER.md（公式にユーザープロファイル専用として定義）
     → Claude系: CLAUDE.md（プロジェクト設定。ユーザー情報が含まれる場合あり）
     → OpenClaw: IDENTITY.md, SOUL.md（エージェント設定だがユーザー情報を含むことがある）
     → Antigravity: GEMINI.md（AI constitution。ユーザー情報を含むことがある）
     → その他: README.md, AGENTS.md 等
     ※ 上記は2026年3月時点の既知のファイル名。今後新しいプラットフォームが登場した場合でも、
       ステップ2b以降のキーワード探索で対応可能。

  b. 名前にユーザー情報を示唆するキーワードを含むファイル
     → about, profile, me, user, 自己紹介, プロファイル, 自分, who-i-am,
       voice, style, context, preferences, working-rules 等

  c. 名前にコンテキスト情報を示唆するキーワードを含むフォルダ
     → context/, config/, docs/, notes/, ABOUT ME/, profiles/ 等
     → 該当フォルダがあれば中のファイルも確認する

  d. その他の .md / .txt ファイル
     → ファイル数が少なければ（10個以下）、冒頭数行を読んで内容を判断する

ステップ3：候補ファイルの内容を読み取る
  → 自然言語で書かれた内容から、氏名・勤務先・役職・地域等を抽出する
  → 構造化されたフォーマット（YAML、キーバリュー等）も非構造化テキストも両方対応
  → 複数ファイルに情報が分散していれば統合する
  → エージェント設定ファイル（AGENTS.md等）の中にユーザー情報が混在している場合、
    ユーザー情報の部分のみ抽出し、エージェント設定の指示は無視する

ステップ4：取得結果を内部コンテキストに保持する
  → 見つかった情報のみ保持。見つからなかった項目はnullとする
  → この探索結果をユーザーに報告する必要はない（バックグラウンド処理）

ステップ5：情報が見つからなかった場合
  → プロファイルなしモードで続行する
  → 質問は増やさない。対応セットの品質は維持する
```

### 2.4 探索の設計原則

- **ファイル名に依存しない。** `USER.md`（OpenClaw）でも `about-me.md`（Claude慣習）でも `CLAUDE.md` の中の一部でも、ユーザー情報があれば使う
- **プラットフォームを仮定しない。** Claude Cowork、Claude Code、OpenClaw、Antigravity、その他互換環境のいずれでも動作する
- **前方互換性を持つ。** 既知のファイル名リスト（ステップ2a）は補助的な手がかりに過ぎない。未知のプラットフォームでも、ステップ2b〜2dのキーワード探索で対応できる
- **探索コストを最小化する。** ディレクトリ一覧→候補絞り込み→内容確認の順で、不要なファイル読み込みを避ける
- **探索に失敗しても止まらない。** プロファイルは「あればボーナス」。なくてもスキルの全機能が動作する
- **読み取りは起動時に1回だけ。** 情報は実行コンテキストに保持し、何度もファイルを開かない

### 2.5 プロファイル活用のルール

- **プロファイルにある情報は質問しない。** 既に「勤務先：○○株式会社」とわかっているのに「お勤め先は？」と聞くのはNG
- **プロファイルにない情報も無理に聞かない。** 冠婚葬祭の対応に不要な個人情報は取得しない
- **文書への差し込みは自動。** 差出人名・署名・勤務先は、プロファイルから取得できた場合は自動的に文書に挿入する
- **プロファイル情報はガイドテキスト内では引用しない。** ファイル内の差し込みにのみ使用し、チャット上の説明には個人情報を含めない
- **プロファイルが見つからなくても正常動作する。** プロファイルなしモードでは、文書の個人情報欄を空欄＋注記にするだけで、対応セットの品質は維持する

### 2.6 プロファイル情報×立場判定の連携

プロファイル情報は立場判定エンジンの精度を大幅に向上させる。

| プロファイル情報               | 立場判定への効果                                         |
| ------------------------------ | -------------------------------------------------------- |
| 役職が「部長」以上             | → 代表対応の可能性を上げる。部署対応・会社対応を優先提案 |
| 役職が「一般」「なし」         | → 個人対応をデフォルトに                                 |
| 勤務先が小規模（個人事業主等） | → 「会社として」は不要。個人対応のみ                     |
| 居住地域が関西                 | → 地域慣習（香典相場の差異等）を自動適用                 |
| 家族構成に「妻」「夫」あり     | → 連名の香典・ご祝儀を提案候補に含める                   |

## 3. 立場判定エンジン（v2新規）

### 3.1 なぜ立場判定が最重要か

同じ「訃報を受けた」でも、立場によって行動が完全に変わる。

- 「親しい友人の親」→ 通夜参列＋香典＋個人的な弔問
- 「取引先の社長の親」→ 弔電＋供花＋社内確認＋会社としての対応

立場判定なしでは、金額も文面もすべてズレる。

### 3.2 判定する4軸

入力から以下の4軸を推定する。推定できない場合のみ質問（最大1問）。

#### 軸1：自分の立場（誰として対応するか）

| カテゴリ   | 具体例                           |
| ---------- | -------------------------------- |
| 友人・知人 | 学生時代の友人、趣味仲間、ママ友 |
| 職場関係   | 同僚、部下、上司、先輩、後輩     |
| 取引先     | 取引先の担当者、取引先の役員     |
| 親族       | 義理の家族、いとこ、遠縁         |
| 近所・地域 | 隣人、町内会、自治会             |
| 保護者仲間 | PTA、子供の同級生の親            |

#### 軸2：相手との関係（誰に何が起きたか）

| 対象     | 意味                         |
| -------- | ---------------------------- |
| 本人     | 結婚する本人、亡くなった本人 |
| 配偶者   | 本人の夫・妻                 |
| 親       | 本人の父・母                 |
| 祖父母   | 本人の祖父・祖母             |
| 子       | 本人の子供                   |
| 兄弟姉妹 | 本人の兄弟姉妹               |

#### 軸3：距離感

| レベル | 説明                 | 判定手がかり                                     |
| ------ | -------------------- | ------------------------------------------------ |
| 親しい | 個人的に深い付き合い | 「仲の良い」「親友」「よく飲みに行く」           |
| 普通   | 一般的な付き合い     | 特に言及なし（デフォルト）                       |
| 形式的 | 面識が薄い・義理     | 「あまり話したことない」「部署が違う」「形だけ」 |

#### 軸4：自分の役割

| 役割     | 説明                         |
| -------- | ---------------------------- |
| 参列者   | 通常の出席者                 |
| 欠席者   | 出席できない人               |
| 代表者   | 会社代表、部署代表として参加 |
| 主催補助 | 幹事、受付係、スピーチ担当   |
| 弔問のみ | 式には出ず、別途弔問         |

### 3.3 判定の自律ルール

- 4軸すべてが明示されていることは稀。入力から推定できるものは推定し、「○○と解釈しました」と明示する
- 推定できない軸がある場合、最も蓋然性の高いデフォルトを採用し、主要な代替パターンも併記する
- 金額に直接影響する軸（立場×関係性）が不明な場合のみ、1問だけ質問する
- 質問は選択肢形式にする（例：「お相手との関係を教えてください：①親しい友人 ②職場の同僚 ③取引先」）

## 4. イベント種別と分岐（v2大幅改訂）

### 4.1 弔事の細分化

v1では「弔事」を一括りにしていたが、実際には「いつ・どう知ったか」「呼ばれたか」「辞退されたか」で対応が根本的に変わる。

| シーン                           | 分岐条件             | 対応の違い                                   |
| -------------------------------- | -------------------- | -------------------------------------------- |
| 訃報を受けた（式に参列）         | 通夜/葬儀の案内あり  | フル対応セット（香典・服装・作法・文面）     |
| 訃報を受けた（家族葬と言われた） | 「家族葬で行います」 | 参列は控える。弔電・供花・後日弔問の判断     |
| 香典辞退と言われた               | 「香典は辞退」       | 香典以外の弔意の示し方（供花・弔電・手紙）   |
| 後日知った                       | 式の後に訃報を知った | 後日弔問・お悔やみの手紙・香典を送るべきか   |
| SNSで知った                      | Facebook等で知った   | 直接連絡すべきか、コメントで済ませるか       |
| 人づてに聞いた                   | 第三者経由で知った   | 確認の取り方・対応の判断                     |
| 会社として対応する               | 総務・管理職の立場   | 社内通知・香典（会社名義）・供花・弔電の手配 |

### 4.2 慶事の細分化

| シーン                     | 分岐条件       | 対応の違い                                 |
| -------------------------- | -------------- | ------------------------------------------ |
| 結婚式に招待された（出席） | 招待状が届いた | ご祝儀・服装・返信はがき・当日マナー       |
| 結婚式に招待された（欠席） | 出席できない   | 欠席返信の文面・ご祝儀を送るか・電報       |
| 入籍報告を受けた（式なし） | 式をしない報告 | 祝い方の判断（贈り物・メッセージ・食事会） |
| 会費制の結婚式             | 「会費制です」 | ご祝儀不要・会費の払い方・別途祝いの判断   |
| 二次会のみ招待             | 二次会だけ     | 会費・服装・プレゼントの判断               |
| 出産報告を受けた           | 出産の連絡あり | 出産祝いの時期・相場・品物                 |

### 4.3 季節行事（v2.1で実装済み）

| シーン | 分岐条件 | 対応の違い |
| ------ | -------- | ---------- |
| お中元 | 夏の贈り物 | 時期（7月上旬〜8月中旬、地域差あり）、品物選び＋送り状 |
| お歳暮 | 年末の贈り物 | 時期（12月上旬〜20日）、品物選び＋送り状 |
| 年賀状 | 年始の挨拶 | 文面、喪中確認（→喪中はがき or 寒中見舞い）、投函時期 |

### 4.4 お祝い（v2.1で実装済み）

| シーン | 分岐条件 | 対応の違い |
| ------ | -------- | ---------- |
| 快気祝い | 病気回復のお返し | 本人→周囲。お見舞いの半返し〜1/3。消え物が定番 |
| 入学祝い | 入学・入園 | 金額は学校レベル×関係性で変動。お返し＝内祝い |
| 七五三 | 3/5/7歳のお祝い | 親族間が一般的。親は準備側（神社＋食事会） |
| 還暦祝い | 60歳の祝い | 赤いもの贈呈の慣習。古希(70)/喜寿(77)/米寿(88)も同フロー |

### 4.5 未実装（Phase 3）

新築祝い、昇進祝い、出産祝い（詳細フロー）

## 5. 出力仕様：「対応セット」（v2大幅改訂）

### 5.1 設計方針の転換

v1は「ガイド＋文書ファイル」という構成だった。v2では「対応セット」という概念に転換する。

**対応セットとは：** ある場面でユーザーが必要とする「情報・判断・文面・行動指示」の全てをパッケージ化したもの。

### 5.2 対応セットの構成要素（7層）

1つのリクエストに対し、以下の7層で出力する。すべてを毎回出すわけではなく、場面に応じて必要な層を自動選択する。

#### 層1：まず確認すべきこと

- 宗派、式の形式、日時、場所など、対応前に確認が必要な情報を整理
- 「まだ確認していないなら、これを聞いてください」のリスト

#### 層2：今すぐやるべき行動（時系列）

- 時系列のアクションリスト
- 「今日中にやること」「前日までにやること」「当日やること」「後日やること」

#### 層3：金額の判断

- 推奨額＋幅（例：5,000円〜10,000円、推奨は5,000円）
- **なぜその額なのか**（判断理由を明示）
- 高すぎる場合のリスク、低すぎる場合のリスク
- 連名にすべきか、個人か会社名義か
- 現金以外の選択肢（供花・弔電・贈り物）

#### 層4：文面（マルチチャネル）

場面に応じて、以下のチャネル別に文面を生成する。

| チャネル       | 用途例                         | 出力形式               |
| -------------- | ------------------------------ | ---------------------- |
| メール（正式） | 欠勤連絡、弔意メール、お礼     | .docxファイル          |
| LINE・チャット | 友人への連絡、カジュアルな弔意 | テキスト（チャット内） |
| SMS            | 短い弔意、確認連絡             | テキスト（チャット内） |
| 社内チャット   | Slack/Teams向けの連絡          | テキスト（チャット内） |
| 弔電・祝電     | 電報サービス向け文面           | .docxファイル          |
| スピーチ原稿   | 結婚式・葬儀でのスピーチ       | .docxファイル          |
| 挨拶状         | 喪中はがき、香典返し挨拶状     | .docx + .pdf           |

#### 層5：口頭セリフ集（v2新規）

文書だけでなく「その場で何を言うか」をカバーする。

| 場面                           | セリフ例の種類       |
| ------------------------------ | -------------------- |
| 受付で                         | 記帳時の一言         |
| 焼香の前後                     | 遺族への声かけ       |
| 通夜振る舞いで                 | 席での会話の切り出し |
| 久しぶりの遺族に               | 声のかけ方           |
| 結婚式の受付で                 | 祝辞の一言           |
| 取引先に電話で弔意を伝える     | 電話の切り出し〜締め |
| エレベーターで偶然遺族に会った | 短い声かけ           |

各セリフは「そのまま言える形」で出力する。解説は最小限にし、丸暗記できる長さにする。

#### 層6：マナー・服装・持ち物

- 服装チェックリスト（印刷用 .pdf）
- 持ち物リスト
- 当日の流れ（タイムライン）
- 作法の要点（焼香・玉串奉奠・献花の手順。宗派別）

#### 層7：NG警告（v2強化）

生成した文面のチェックだけでなく、行動全般のNG警告を出す。

**言葉のNG：**

- 忌み言葉（重ね言葉、不吉な表現）
- カジュアルすぎる表現
- 不適切な件名
- 宗派に合わない表現（浄土真宗に「ご冥福」等）

**行動のNG：**

- 香典の金額が偶数・4/9を含む
- 新札を使う（弔事）/ 旧札を使う（慶事）
- 不要な詮索（死因を聞く等）
- 返信の遅れ（招待状の返信期限）
- 式中の写真撮影
- 毛皮・革製品の使用（弔事）

## 6. 会社関係モード（v2新規）

### 6.1 なぜ独立モードが必要か

会社関係の冠婚葬祭は、個人の判断だけでは完結しない。「個人として」「部署として」「会社として」の3レイヤーがあり、それぞれの対応が異なる。

### 6.2 3レイヤーの判定

| レイヤー | 判定条件               | 対応内容                                   |
| -------- | ---------------------- | ------------------------------------------ |
| 個人対応 | 個人的な関係がある場合 | 個人名義の香典・ご祝儀、個人的なメッセージ |
| 部署対応 | 同じ部署の場合         | 連名の香典、部署としての弔電・供花         |
| 会社対応 | 役員・重要取引先の場合 | 会社名義の香典・供花・弔電、総務への連絡   |

### 6.3 会社関係で追加出力するもの

- 総務・上司に確認すべき事項リスト
- 社内通知文（部署内メール・チャット用）
- 会社名義の場合の手配フロー
- 個人と会社の両方で対応する場合の切り分けガイド

### 7.1 メインフロー

    │  ├─ 候補ファイルの内容を読み取り、ユーザー情報を抽出
    │  ├─ 見つかった → 氏名・勤務先・役職・地域・家族構成を内部保持
    │  └─ 見つからない → プロファイルなしモードで続行（質問は増やさない）
    ▼
[1] イベント種別の判定
    │  ├─ 弔事 → 細分化（参列/家族葬/香典辞退/後日/会社対応）
    │  ├─ 慶事 → 細分化（出席/欠席/式なし/会費制）

    │  ├─ お祝い → 快気祝い/入学祝い/七五三/還暦祝い

    ▼
[2] 立場判定エンジン（4軸）＋プロファイル情報で精度向上
    │  ├─ 自分の立場（プロファイルの役職・勤務先から補完）
    │  ├─ 相手との関係
    │  ├─ 距離感
    │  └─ 役割（プロファイルの役職で代表対応の可能性を判定）
    │  推定結果を「○○と解釈しました」で明示
    │  金額に影響する軸が不明 → 1問だけ質問
    ▼
[3] タイミング分岐の判定
    │  ├─ 事前に知った
    │  ├─ 当日に知った
    │  ├─ 後日知った

    │  ├─ ng_words.md（NGワード辞書＋代替表現）
    │  └─ company_mode.md（会社対応時のみ）
    ▼
[6] 🔍 Web検索（変動情報のみ）（v3.0新規）
    │  金額の最新相場、トレンド、年固有情報のみWebで取得
    │  Web検索が失敗してもreferences/の知識で完全な対応セットを返せる
    ▼
[7] 対応セットの構成決定
    │  場面に応じて7層のうち必要な層を自動選択

    ▼

    │  ├─ チャット内テキスト（層1〜3, 5, 7）

### 7.2 困難を自律的に乗り越えるポイント

| 困難                               | 自律的な対処                                                                                                                                                                           |
| ---------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| プロファイルが見つからない         | プロファイルなしモードで続行。文書の個人情報欄は空欄＋注記。対応セットの品質は維持                                                                                                     |
| プロファイルの情報が古い/不完全    | 存在する情報のみ使用。不足分は推定 or 空欄。古い可能性がある情報（役職等）は「プロファイルに○○とありましたが、現在も同じですか？」と確認しない（冠婚葬祭の緊急時に余計な質問をしない） |
| プロファイルと入力が矛盾           | 入力を優先。例：プロファイルに「個人事業主」とあるが「会社として弔電を」と言われた場合、入力に従う                                                                                     |
| 入力が曖昧（「香典いくら？」のみ） | プロファイルから立場を補完。それでも金額判断に足りなければ1問だけ質問（選択肢形式）                                                                                                    |
| 宗派が不明                         | 仏式（一般的）で出力＋「浄土真宗の場合はここが違います」を注記                                                                                                                         |
| 家族葬と言われた                   | 「参列は控える」をデフォルトに。弔意の示し方（弔電・手紙・後日弔問）を提案                                                                                                             |
| 香典辞退と言われた                 | 香典以外の選択肢を提示（供花・弔電・手紙・後日の弔問）                                                                                                                                 |
| 後日知った                         | 経過日数に応じた対応を分岐（1週間以内/1ヶ月以内/それ以上）                                                                                                                             |
| 会社か個人か曖昧                   | プロファイルの役職を参考にしつつ、両方のパターンを出力                                                                                                                                 |
| 地域慣習の差                       | プロファイルの居住地域があれば自動適用。なければ全国標準                                                                                                                               |
| NGワードの混入                     | 生成後に自動スキャン。検出時は代替表現に置換し、「ここを修正しました」とログ                                                                                                           |
| 複数文書間の矛盾                   | クロスチェックで金額・宛名・日付の不一致を自動修正                                                                                                                                     |

### 8.2 ハイブリッド知識戦略（v3.0新規）

SKILL.mdには判定ロジックと出力フロー制御のみ記載。詳細知識はreferencesから動的に読み込み、最新情報はWeb検索で補完。

| 知識の種類 | 取得元 | 理由 |
| ---------- | ------- | ---- |
| のし袋・焼香・服装・口頭セリフ | `references/etiquette.md` | 言語的・宗教的タブーであり時代で変わらない |
| NGワード・代替表現 | `references/ng_words.md` | 同上 |
| 会社対応のフレーム | `references/company_mode.md` | 組織構造に基づく不変のフロー |
| 金額の最新相場 | Web検索 | 物価・社会変化で変動 |
| 人気の贈り物・トレンド | Web検索 | 毎年変わる |
| 年固有情報（干支等） | Web検索 | 毎年変わる |
| 地域の最新事情 | Web検索 | 地域慣習は徐々に変化 |

**安全設計:** Web検索が全滅しても、references/ の知識だけで完全な対応セットを返せる。「情報が見つかりませんでした」とは絶対に言わない。

## 10. 評価基準（v2改訂）

### 10.1 定量評価

| 指標                         | 基準                                                                   |
| ---------------------------- | ---------------------------------------------------------------------- |
| NGワード混入率               | 0%（忌み言葉が1つでも混入していたら不合格）                            |
| 行動NGの警告漏れ             | 0%（既知のNG行動が警告なしで通過したら不合格）                         |
| 金額相場の正確性             | 一般的な相場資料との乖離が±20%以内                                     |
| 金額判断理由の有無           | 100%（金額提示時に必ず理由が付いていること）                           |
| ファイル生成成功率           | 100%（全ファイルが指定形式で正常に開けること）                         |
| 宗派の一貫性                 | 1出力内で宗派表現が混在していないこと                                  |
| 文書間の整合性               | 金額・宛名・日付・宗派が全出力で一致                                   |
| 立場判定の明示率             | 100%（推定結果を必ず「○○と解釈しました」で表示）                       |
| 口頭セリフの提供率           | 対面場面では100%（参列・弔問・受付がある場面）                         |
| プロファイル活用率           | プロファイルに情報がある場合、文書の差出人・署名に100%自動挿入（v2.2） |
| プロファイルなし時の正常動作 | プロファイルなしでも対応セットが正常出力されること（v2.2）             |
| 不要質問の抑制               | プロファイルに既にある情報を再質問していないこと（v2.2）               |

### 10.2 定性評価

| 指標                     | 基準                                                                                                                     |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| 包括性                   | 「他にもやることあったっけ？」と思わせない                                                                               |
| 安心感                   | 「これで大丈夫」と感じられるトーン                                                                                       |
| 即行動可能性             | 出力を見たら次の行動に迷わない                                                                                           |
| 口頭セリフの自然さ       | そのまま言っても不自然でない日本語                                                                                       |

### 11.2 SKILL.mdの構成方針

- SKILL.md本体は **500行以内** に収める（現在356行）
- 本体に含めるもの：プロファイル検索手順、トリガー判定、立場判定エンジンの4軸ロジック、イベント分岐表、ハイブリッド知識戦略、対応セット構成ルール、ファイル生成手順、クロスチェック手順
- **スキル起動時の最初のアクションとして、ワークスペースの自律探索によるユーザー情報取得を行う指示を明記する（特定のファイル名はハードコードしない）**
- 冠婚葬祭の詳細知識は全て `references/` に分離（3ファイルに統合）
- 会社関係モードのロジックは `references/company_mode.md` に分離
- 口頭セリフ集は `references/etiquette.md` 内に統合（弔事・慶事横断で一括参照可能）

### 11.3 制約事項

- 法的助言は行わない（相続・遺言→「専門家に相談してください」）
- 宗教的な価値判断は行わない（全宗派を等価に扱う）
- 金額は「一般的な相場」として提示。「絶対的な正解」とは表現しない
- 個人情報は文書内でのみ使用。ガイド部分には含めない
- 医療・介護の助言は行わない

## 13. 競合優位性

| 既存の代替手段   | このスキルの優位点                                                 |
| ---------------- | ------------------------------------------------------------------ |
| Google検索       | 情報が断片的。10サイト巡回が必要。このスキルは一言で全部出る       |
| マナー本         | 手元にない。検索性が低い。「自分のケース」にマッチしない           |
| ChatGPTへの質問  | テキスト回答のみ。ファイル出力・口頭セリフ・NG警告・立場判定がない |
| マナーサイト     | 広告だらけ。家族葬・香典辞退など現代的な分岐に弱い                 |
| 会社の総務に聞く | 週末・夜間は聞けない。聞くこと自体が気まずい                       |

**このスキルだけが持つ独自価値：**

1. **ワークスペースを自律探索してユーザー情報を取得し、質問ゼロで最適な対応セットを出力** — プラットフォームを問わず、どんなファイル構成でもAIが自分で情報を見つける
2. 立場判定→金額判断理由→文面→口頭セリフ→NG警告を **一気通貫で** 出力
3. 「家族葬」「香典辞退」「式なし婚」「後日知った」など **現代的な分岐** に対応
4. メール・LINE・チャット・口頭という **全チャネル** をカバー
5. 判断に理由がある（「相場は○円です」ではなく「○○だから○円が適切です」）
